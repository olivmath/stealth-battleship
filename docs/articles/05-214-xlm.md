### A arquitetura final

Depois de todos os erros, pivots e descobertas, a arquitetura ficou assim:

```
                    Gera provas ZK
                    (NoirJS + bb.js WASM)
                          |
    [Cliente Web]  -------|
         |                |
         | Socket.io      |
         | (Ed25519)      |
         v                v
    [Backend Express] ----------> [Supabase]
         |                         (histórico, stats)
         |
         | 4 TXs por partida
         v
    [Soroban Testnet]
         |
         |-- BattleshipContract (nosso)
         |     |
         |     |-- verify_board(proof) --> VerifierContract
         |     |-- open_match(p1, p2) --> GameHub.start_game()
         |     |-- close_match(proof) --> VerifierContract
         |     |                      --> GameHub.end_game()
         |
         |-- UltraHonk Verifier (BN254 nativo, Protocol 25)
         |-- Game Hub (Stellar ecosystem)
```

Três camadas com responsabilidades claras:

1. **Cliente**: gera provas ZK localmente (nunca envia posições dos navios)
2. **Backend**: gerencia partida PvP, verifica Provas ZK, submete on-chain
3. **Blockchain**: verificação final e trustless, registro imutável

### As 5 lições que eu pagaria para aprender antes

#### 1. Teste a integração mais arriscada primeiro

Eu comecei pelo jogo single-player. 14 telas bonitas, animações, haptics. Duas semanas de polish. Quando tentei integrar bb.js no React Native... não rodava. O app mobile virou single-player only, e tive que fazer um cliente web do zero.

Se eu tivesse gasto 2 horas no primeiro dia testando `import { UltraHonkBackend } from '@aztec/bb.js'` no React Native, teria ido direto para web e economizado dias.

**Regra: a primeira coisa que você faz num projeto é testar a parte que pode matar o projeto.**

#### 2. Fees de blockchain são dinâmicos — não chute

Coloquei 214 XLM (max u32 stroops) como fee porque "assim não falha". E realmente não falhava. Mas eu também não sabia quanto estava pagando de verdade.

Quando finalmente olhei os custos reais via Horizon API:

```bash
curl "https://horizon-testnet.stellar.org/transactions/{hash}" | jq '.fee_charged'
```

Descobri que a primeira TX custava 5.5 XLM (cold start) e as seguintes 0.03 XLM. O fee max não é o que você paga — é o teto que você aceita. O Soroban simula a transação e cobra o custo real.

A solução final: usar o `minResourceFee` da simulação:

```typescript
const simResult = await server.simulateTransaction(tx);
const minFee = simResult.minResourceFee;
console.log(`${method} custo real: ${minFee/1e7} XLM`);
```

**Regra: nunca ignore os custos. Logue, entenda, otimize.**

#### 3. Sequence numbers são estado mutável — cuidado

Tentei otimizar o fee reconstruindo a transação com o valor correto da simulação. Mas o `TransactionBuilder` incrementa o sequence number da conta internamente. Ao reconstruir, o sequence ia para N+2 quando a simulação esperava N+1.

```
TX simulada: sequence N+1 ✓
TX reconstruída: sequence N+2 ✗ → ERROR
```

Parece óbvio olhando depois. Mas no momento, estava enterrado em 50 linhas de async code e o erro era um genérico "status=ERROR" sem mensagem descritiva.

**Regra: efeitos colaterais em construtores são armadilhas. Leia o código-fonte do SDK, não só a documentação.**

#### 4. ZK não é mágica — é trade-off

Zero-knowledge proofs são poderosas, mas têm custos claros:

| Aspecto | Trade-off |
|---------|-----------|
| **Tamanho do circuito** | Mais verificações = mais constraints = Prova ZK mais lenta |
| **Inputs fixos** | Circuitos não suportam arrays de tamanho variável — você define o máximo e preenche o resto |
| **Tempo de geração** | board_validity: ~2s. turns_proof: ~12s. Aceitável num jogo, inviável em real-time |
| **Compatibilidade** | bb.js precisa de WASM pesado — não roda em todo ambiente |
| **Curva de aprendizado** | Pensar em "circuitos" é diferente de pensar em "funções" |

O `turns_proof` gasta 12 segundos porque itera 100 ataques x 5 navios x 5 células por navio = 2500 verificações de hit. Cada verificação vira constraints no circuito. Não tem como "otimizar" sem reduzir o que o circuito verifica.

A decisão de verificar `shot_proof` só no backend (não on-chain) foi trade-off consciente: velocidade em troca de confiança parcial. O `turns_proof` no final compensa — ele verifica tudo retroativamente.

#### 5. State rent é invisível até doer

O Soroban cobra aluguel (rent) para manter dados on-chain. Cada pedaço de estado — WASM, chaves de verificação, variáveis — tem um TTL. Na testnet, o TTL padrão do Instance storage é de ~5 dias. Quando expira, o estado fica archived. A próxima chamada precisa pagar **restore**, e o custo é proporcional ao tamanho dos dados.

Nosso contrato Verifier carrega o bytecode UltraHonk (pesado) e duas verification keys. Quando o TTL expirava, a primeira `verify_board` custava **5.54 XLM** em vez de 0.03 XLM — 163 vezes mais cara.

O bug? Nossos contratos faziam `extend_ttl` no Temporary storage (MatchState, com 30 dias de TTL), mas **não faziam no Instance storage** — justamente onde estão os dados mais caros (WASM e VKs). O Instance ficava com o TTL padrão da rede, e expirava silenciosamente.

O fix é uma linha em cada contrato:

```rust
env.storage().instance().extend_ttl(518_400, 518_400); // 30 dias
```

Adicionando isso em `verify_board` ou `open_match`, cada chamada renova o TTL da instância por mais 30 dias. Custo marginal: zero. Custo de não fazer: 5+ XLM toda vez que o contrato fica frio.

Analisando as 202 transações da conta, o padrão era claro: dias sem uso → restore caro → chamadas normais baratas → dias sem uso → ciclo se repete. Um `extend_ttl` teria evitado **~16 XLM** em restores durante o desenvolvimento.

**Regra: entenda o modelo de rent da sua blockchain antes de deployar. O que você não estende, a rede arquiva.**

### O que ZK muda nos jogos

Além das lições técnicas, esse projeto me mostrou algo mais amplo: zero-knowledge proofs resolvem um problema fundamental de jogos digitais.

Em qualquer jogo com informação oculta — poker, Battleship, fog of war, cartas na mão — alguém precisa ser o "árbitro confiável". Historicamente, esse papel cai no servidor. Você confia que o servidor do jogo não está te roubando. E na maioria dos jogos casuais, essa confiança é aceitável.

Mas imagine jogos com stakes reais. Torneios. Dinheiro envolvido. Apostas. De repente, "confiar no servidor" não é mais aceitável. Você quer **Provas ZK**.

Com ZK:
- O jogador posiciona seus navios e gera uma Prova ZK de validade
- Cada jogada é acompanhada de uma Prova ZK de honestidade
- O resultado é verificado matematicamente, não por confiança

Ninguém precisa confiar em ninguém. A matemática é o árbitro.

---

### O Protocol 25 como habilitador

Isso só é economicamente viável por causa do Protocol 25 da Stellar. A verificação de provas UltraHonk depende de operações de curva elíptica BN254 e hash Poseidon2. Antes do Protocol 25, fazer essas operações dentro de um smart contract seria proibitivamente caro — seriam milhões de instruções WASM.

Com o Protocol 25, essas operações são **host functions** — executadas nativamente pelo validator. O custo cai de "impossível" para "0.05 XLM".

Isso abre a porta não só para Battleship, mas para qualquer jogo de informação oculta: poker, Mahjong, Stratego, Cluedo, jogos de cartas. Qualquer mecânica onde "eu sei algo que você não sabe" pode ser protegida por ZK e verificada on-chain.

3. **Dia 2**: fee dinâmico desde a primeira transação, com logs
